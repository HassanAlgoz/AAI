Shell Basics

Why learn terminal commands?

  • Faster for repetitive tasks (rename/copy/search many files)
  • Works the same on local machine, servers, and cloud VMs
  • Easier to automate and document exact steps
  • Needed for many developer tools (git, python, package managers)
  • Useful when GUI is unavailable or too slow

When terminal beats GUI

  • You need to process many files at once
  • You want reproducible steps in notes/scripts
  • You work over SSH on a remote Linux machine
  • You debug environments and paths quickly
  • You chain commands into one workflow

Terminal command anatomy (POSIX style)

Most commands follow this pattern:

command [options] [required-args] [optional-args]

Examples:

ls -la /tmp
cp -r src_dir backup_dir
head -n 5 notes.txt

Required vs optional arguments

  • Required arguments: command fails without them
    • mv old.txt new.txt
  • Optional arguments: command still works without them
    • ls vs ls -la
  • Flags/options change behavior
    • -a, -l, -r, --help

Single-dash and double-dash flags

  • Short flags: -a, -l, -n 10
  • Long flags: --all, --help
  • Some commands allow combining short flags: ls -la
  • Always check exact syntax in docs (man command or command --help)

cd (change directory)

  • Purpose: move into another folder
  • macOS/Linux: cd, cd .., cd /path/to/dir
  • Windows cmd: cd, cd .., cd C:\path\to\dir
  • Common pattern:
pwd
cd ..
pwd

ls (list directory contents)

  • Purpose: show files/folders in current or target directory
  • macOS/Linux: ls, ls -la, ls /tmp
  • Windows cmd equivalent: dir
  • Useful flags:
    • -l long format
    • -a include hidden files

pwd (print working directory)

  • Purpose: show absolute path of current location
  • macOS/Linux: pwd
  • Windows cmd equivalent: cd (without args shows current path)

which (find executable path)

  • Purpose: show where a command is found in PATH
  • macOS/Linux: which python, which git
  • Windows cmd equivalent: where python
  • Common Pattern:
    • Program paths affect program behavior:
      • based on config files
      • based on nested files (like libraries)
    • Used alongside python --version and git --version to also know which version are we running

whoami (current user)

  • Purpose: show active username
  • macOS/Linux: whoami
  • Windows cmd equivalent: whoami
  • Used to know our access control scope:
    • Admin / Super-user has the highest level of privleges
    • others have different permissions

man (manual pages)

  • Purpose: open built-in docs for commands
  • macOS/Linux: man ls, man cp
  • Windows cmd equivalent: help (or command /?, e.g. dir /?)
  • Other ways to get help:
    • explainshell.com
    • tldr.sh
    • Your own notes

cat (print file contents)

  • Purpose: print file to terminal (and concatenate files)
  • macOS/Linux: cat file.txt, cat a.txt b.txt
  • Windows cmd equivalent: type file.txt

less (read large files page-by-page)

  • Purpose: scroll through long text safely
  • macOS/Linux: less big.log
  • Navigation: Space next page, b previous page, q quit
  • Windows cmd equivalent: more file.txt

touch (create empty file / update timestamp)

  • Purpose: create a file quickly if missing
  • macOS/Linux: touch notes.txt
  • Windows cmd equivalent: type nul > notes.txt (create empty file)

head (first lines of a file)

  • Purpose: preview beginning of a file
  • macOS/Linux: head file.txt, head -n 20 file.txt
  • Windows cmd equivalent: more +1 file.txt (rough equivalent)

tail (last lines of a file)

  • Purpose: preview end of a file
  • macOS/Linux: tail file.txt, tail -n 20 file.txt
  • Windows cmd equivalent: more -20 file.txt (or type file.txt | more, scroll to end)

mkdir (make directory)

  • Purpose: create directories
  • macOS/Linux: mkdir data, mkdir -p a/b/c
  • Windows cmd equivalent: mkdir data

file (identify file type)

  • Purpose: detect file content/type
  • macOS/Linux: file report.pdf, file script.sh
  • Windows cmd equivalent: no exact built-in direct equivalent in cmd

mv (move or rename)

  • Purpose: move file/folder or rename it
  • macOS/Linux: mv old.txt new.txt, mv file.txt docs/
  • Windows cmd equivalent: move (and ren for rename)

cp (copy files/folders)

  • Purpose: duplicate files/directories
  • macOS/Linux: cp a.txt b.txt, cp -r src_dir backup_dir
  • Windows cmd equivalent: copy (files), xcopy (folders)

find (search files by criteria)

  • Purpose: locate files/directories by name/type/pattern
  • macOS/Linux: find . -name "*.py", find . -type d
  • Windows cmd equivalent: dir /s /b *.py

cmp (compare two files byte-by-byte)

  • Purpose: quickly check if files differ
  • macOS/Linux: cmp file1.bin file2.bin
  • Windows cmd equivalent: fc /b file1.bin file2.bin

df (disk free space)

  • Purpose: show filesystem usage
  • macOS/Linux: df, df -h
  • Windows cmd equivalent: wmic logicaldisk get size,freespace,caption

Mini workflow example

pwd
mkdir -p demo/data
cd demo
touch notes.txt
cp notes.txt data/notes_copy.txt
ls -la data
df -h

Practice: classify command parts

For each command, identify command, flags, required args, optional args:

head -n 5 README.md
cp -r src_dir backup_dir
find . -name "*.qmd"

Opencode

  • Opencode is an AI coding assistant you can run from your shell
  • It works well for repo exploration, editing, and command execution workflows
  • Treat it like a pair-programmer: give clear goals, files, and constraints
  • Always review generated changes before committing

Opencode installation

Visit the website:

curl -fsSL https://opencode.ai/install | bash

Opencode quick start flow

  1. Open your project in terminal
  2. Start Opencode
  3. Give a focused prompt with file paths and expected outcome
  4. Review diffs, run tests, then commit
cd <project>
opencode

Prompting Opencode effectively

  • Be specific about:
    • target files (path/to/target/file)
    • required output (e.g., exercise questions)
    • constraints (keep style consistent, avoid breaking existing sections)
  • Ask for verification steps:
    • “run checks”
    • “show me the diff”
    • “summarize what changed”

Security concerns with agents that run commands

  • Prompt injection: malicious instructions in files/URLs that trick the agent
  • Data exfiltration: agent can leak secrets (API keys, tokens, private data) via logs/output/network
  • Untrusted code execution: agent may suggest dangerous commands (delete files, download/run binaries)
  • Filesystem changes: agent can modify anything your user can access
  • Supply-chain risk: installing tooling from scripts/URLs may execute attacker-controlled code
  • Over-trusting generated output: agent can “hallucinate” commands or diffs that look correct but are wrong

Mitigations (what to do safely)

  • Never paste secrets into prompts; use env vars or secret stores when needed
  • Run as an unprivileged user; avoid sudo unless absolutely necessary
  • Review the diff first; only then run tests and accept changes
  • Prefer sandboxing: container/VM, restricted filesystem, and limited network access
  • Use least-privilege permissions and allowlisting where possible (e.g., only run known-safe commands)
  • Keep auditability: require “show commands” and capture logs before executing

Wrap-up

  • Terminal is about speed, repeatability, and control
  • You now know core navigation and file-management commands
  • You can read docs, pass flags/arguments, and inspect your system
  • You can install and run Python CLI tools with uv
  • You can use Opencode to accelerate terminal-first coding tasks
