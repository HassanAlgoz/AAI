Model Evaluation

Learning Objectives

In this lecture we learn to:

  1. Establish a baseline for learning
  2. Understand when more data helps (and when it does not)
  3. Detect over-fitting vs under-fitting and reason about generalization
  4. Estimate generalization with proper train/validation/test splits and cross-validation

Frame the lecture as a toolkit for evaluation: (1) what we optimize vs what we measure, (2) how to catch obvious failures early with baselines, and (3) how to estimate real-world performance without data leakage.

How to verify the model is learning?

  • Core sanity check:
    • the model should be better than a guess
    • and certainly better than random predictions
  • If it is not, common causes include:
    • wrong model
    • unhelpful features
    • wrong hyperparameters
    • class imbalance issues

Deliver the mindset: if evaluation isn’t better than chance, don’t jump straight to complex models—first check whether the pipeline and settings let the learner extract signal from X.

DummyClassifier: baselines that ignore inputs

DummyClassifier implements simple “rules of thumb” baselines:

  • constant: always predicts a provided constant label
  • most_frequent: always predicts the most frequent training label
  • stratified: random predictions respecting the class distribution

Key point: in all cases, predict() ignores input features.

Reinforce why baselines matter: they represent the performance you get without learning from the input. Your real model should beat them; otherwise, something is wrong (features, targets, or evaluation setup).

Compare dummy vs real model (classification)

from sklearn.dummy import DummyClassifier

dummy_clf = DummyClassifier(strategy="most_frequent")
dummy_clf.fit(X_train, y_train)

print("   Dummy Classifier Score:", dummy_clf.score(X_test, y_test))
print("Logistic Regression Score:", clf.score(X_test, y_test))
  • Your trained model should beat the dummy baseline
  • If accuracy is near baseline, something is likely wrong

In delivery, say: “first we establish a yardstick.” Then interpret the gap: a small gap means the model is not extracting useful information; a large gap suggests the learning problem is well-posed.

DummyRegressor: regression baselines

DummyRegressor implements simple rules:

  • mean: always predicts the mean of training targets
  • median: always predicts the median of training targets
  • quantile: always predicts a user-provided quantile
  • constant: always predicts a provided constant value

Key point: these strategies ignore input features during predict().

Transition the baseline idea from classification to regression: the same logic applies—before trusting a model’s absolute score, confirm it beats a “predict the mean/median” strategy.

What does “more data” mean for a model?

More data helps only if:

  1. It fills gaps: new samples cover regions where the model lacks information
  2. The model is capable: the learning algorithm can represent the real pattern
  3. Data quality matters: avoid “garbage in - garbage out”

Our goal is new information, not just more data points.

Set expectations: collecting more data is an investment, not a guarantee. Performance improves only when at least one of these conditions is the limiting factor.

Reinforce: if the model family cannot represent the true pattern, collecting more samples just makes the wrong bias more “confident.” The fix is to increase model capacity or add better features.

Over-fitting and under-fitting

Both regression and classification models can:

  • under-fit (too general)
  • fit just right
  • over-fit (too specific)

Make the “diagnostic loop” explicit: you’re not just fitting a model, you’re checking whether it generalizes. Training error alone cannot tell you whether the model is too simple or too complex.

Frame this as “complexity vs generalization.” Under-fitting misses the real structure (too much bias). Over-fitting captures idiosyncrasies/noise (too much variance). The optimal region is the one that would perform best on unseen data.

For regression, interpret “wiggliness” as capacity. Under-fit curves are too rigid and miss patterns; over-fit curves follow noise. The goal is not to minimize training error only, but to get the curve that extrapolates/generalizes best.

Bias-variance Tradeoff

  • Generalization:
    • how well a model fits data it has not seen during training
  • Bias vs Variance (sources of error):
    • Bias: error from being too simple
      • corresponds to under-fitting
    • Variance: error from being too sensitive to training data
      • corresponds to over-fitting
  • A good model strikes a balance between both

Give a one-sentence operational definition: generalization is performance on unseen data. Then map failures to bias (systematic under-explanation) and variance (sensitive over-adaptation).

Cross-validation: reliable estimates

  • With a single train-test split: the error estimate is not reliable.
  • Cross-validation uses k-folds (splits) and averages results:
    • fit on training folds
    • test on validation fold

Say why CV exists: a single split can be misleading if the test fold happens to be unusually easy or hard. CV averages over multiple splits and helps you detect over-fitting.

Comparing models more reliably

Point at the variability: if the “best” model also has consistently high test scores across folds, it’s a safer choice. Large swings across splits is a sign of sensitivity to the sampled data.

Train-test Errors Gap

  • x-axis maybe: model complexity, dataset size, or even training time.
  • Gap analysis
    • Widening gap: over-fitting.
    • Both high: under-fitting.
    • Both low: perfect model (check if the data is representative of the real probelm).

Reading learning curves: pay attention to whether the gap between train and test shrinks (variance reduction via more data) or whether both curves improve slowly (bias/under-fitting). When the test curve plateaus, new data often becomes diminishing returns.

Train / Validation / Test splits

Training set: fit parameters.

Validation set: select intermediate models/hyperparameters.

Testing set: final unbiased performance check.

Warning

If you use the testing set to make decisions, you effectively turn it into a validation set. Never let the model “see” test information.

Note

Another cross-validation procedure can be used with testing set. If you use CV in both train-validation split, and then again on testing set, you will have \(n \times m\) fit-and-score runs; which needs a lot of compute. Hence, less people do this especially when the dataset is very big.

Delivery cue: show a “clean loop.” Train on training data, tune on validation, and keep the test set locked until the final evaluation. The warning is about leakage through decision-making.

Summary: What to remember

  • Loss is a training objective; metrics are for evaluation and comparison
  • Verify learning by comparing against dummy baselines
  • More data helps when it:
    • fills gaps
    • matches model capability
    • is high quality
  • Diagnose generalization:
    • over-fitting vs under-fitting
    • bias vs variance
  • Use correct evaluation:
    • train/validation/test splits
    • cross-validation for robustness (including time-aware CV when needed)

Final wrap-up: if you remember one workflow, it’s this—(1) choose the right metric for the problem, (2) sanity-check with baselines, (3) diagnose with learning curves/bias-variance, and (4) estimate generalization with proper splits (time-aware when needed).
